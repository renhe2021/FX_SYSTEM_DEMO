# FX Desk Presentation Skill — 外汇交易台演示文稿设计规范

## 用户画像

- **身份**：腾讯外汇交易台员工，非技术岗但会写代码
- **日常**：报表、定价、策略、市场情报、跟产研提需求推功能上线
- **工具环境**：CodeBuddy + MCP + Skill，已在实际工作中广泛使用
- **性格特点**：说话直，不喜欢绕弯子，反感包装过度的东西

---

## 核心审美原则：去 AI 味

### 什么叫「AI 味」

AI 生成的内容有几个典型特征，用户极度反感：

1. **辅助文字堆砌** — 每个标题下面跟一段解释性小字，像产品说明书
2. **副标题过多** — "方法论 · 实践 · 回顾"，"选事件 → 五阶段响应 → 定价/风控双轨展示"
3. **Emoji 泛滥** — 📊📝📚💡🎯💥 满屏表情符号
4. **金句/鸡汤句** — 每页底部都有一句"总结性"的话
5. **过度结构化** — bullet point、编号、分类，把简单的事搞复杂
6. **套话连接词** — "综上所述"、"值得注意的是"、"这让我意识到"

### 正确做法

- **Slide 上能少写就少写**，细节留给口头说
- **截图驱动** — 一张截图顶十句描述
- **标题即内容** — 标题本身就够说明问题，不需要再解释
- **不用 emoji** — 除非是刻意调侃（而且极少）
- **不写金句** — 如果非要有，放在演讲稿里口头说，别写在 slide 上

## Slide 设计规范

### 整体框架

- **19 页 HTML slides**，1920×1080，CSS 控制样式，JS 控制翻页
- **三幕叙事结构**：蓝色（分析与交易）→ 绿色（文字与表达）→ 紫色（知识沉淀）
- 颜色体系：`#0052D9`（蓝）、`#2BA471`（绿）、`#7B61FF`（紫）

### 页面类型规范

| 类型 | 规范 | 示例 |
|------|------|------|
| **Section Divider** | 只留编号圆圈 + 标题，**不加小字描述** | P3/P9/P14 |
| **LIVE/DEMO 页** | 只留 Badge + 标题 + 链接，**不加副标题** | P7/P12 |
| **内容页** | 标题 + 核心内容，**不加底部金句栏** | P4/P5/P6... |
| **工具全景页** | Tag 列表 + 截图，**不加底部总结语** | P8/P13 |
| **Agent 卡片页** | 截图驱动，**不加文字描述** | P15 |

### 文字精简原则

- 一个意思用一句话说完，不用两句
- 能删的就删，留着让演讲者口头补
- `<br>` 换行不超过 1 处，超过说明文字太多了
- Tag 内文字控制在 15 个汉字以内

## 演讲稿设计规范

### 调性

- **像跟朋友聊天**，不是在读稿
- 不用 "首先/其次/最后" 这种套路，用 "第一块/第二块" 或自然过渡
- 可以有口语连接词："说个具体的"、"好，下一页"、"补一句"
- 适当加入自嘲和真实感受，但不要刻意搞笑

### 结构规范

- 每页标注预估时长，总时长 20-25 分钟（含 Demo）
- Demo 页只写 *（现场演示。自然讲解即可。）*，不写台词
- 数据、参数名、系统名用准确的术语，不要模糊化
- 翻车故事要诚实讲，不要包装成"有意义的探索"

### 口语化检查清单

- [ ] 每段话能不能一口气念完？太长就拆
- [ ] 有没有书面语混进来？（"综上"、"此外"、"不难发现"）
- [ ] Bullet point 格式改成自然段落了吗？
- [ ] 过渡是否自然？（不要生硬跳转）
- [ ] Demo 前有没有一句"说了这么多不如直接看"这种引入？

## Word 导出规范

- 使用 `docx-js` 库，A4 纸，微软雅黑字体
- 每页用蓝色带底线标题标注页码和时长
- 正文为可直接念的口语化文字
- 舞台指示用灰色斜体
- **注意**：JS 中使用中文引号 `""` 会导致 SyntaxError，需用 `\u201C`/`\u201D` 转义或改用单引号
